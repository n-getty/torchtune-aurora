# Qwen3-30B-A3B EP=8 — v10 Series, 2-Node (2026-04-30)

PBS jobs 8460378 and 8461394 (capacity, 2 nodes). Continuation of the v8
3-node series (`qwen3_ep_v8_3node_20260430.md`) and the v9 grad-release
helper (`project_qwen3_ep_v9_helper.md`).

## Goal

Validate the v9 `_ep_release_fsdp_unsharded_grads` helper end-to-end at
EP=8 / dp_replicate=1 / dp_shard=8 on a single training node, then push to
multi-step + multi-chunk to characterize the production envelope.

## What we learned

- **v9 helper works at EP=8.** `accumulate_into_grad=True` fires correctly
  for all 97 FSDPParamGroups → gloo dp_shard PG → ~25s per release. The
  v59 `reduce_grads=False` no-op (which was silently leaving the unsharded
  grad pool resident across chunks AND across steps) is fully replaced by
  the explicit reduce-scatter.

- **AdamW fp32 state is the second wall.** With the v9 helper releasing
  unsharded grads correctly, plain `torch.optim.AdamW` still pinned
  PRE-STEP-1 to 29.99 GiB (vs PRE-STEP-0 = 7.19 GiB). The +22.8 GiB delta
  is fp32 momentum + variance materialized on the first `optimizer.step()`
  and never released. Step-2 fwd then ran into `l0_free < 0.1 GiB`. v10c
  and v10d (same crash at minimum activation, G=1) confirm this is
  optimizer state, not unfreed grads or activation pressure.

- **AdamWBf16 unblocks multi-step.**
  `torchtune.dev.bioreason.optim.AdamWBf16` keeps moments on CPU in BF16
  and only ships a per-param delta to the GPU. PRE-STEP-1 dropped from
  29.99 → 15.77 GiB on v10e — restoring ~14 GiB of L0 headroom. v10f
  (G=4 NSTEPS=3) then ran clean for all 3 steps with PRE-STEP-2 (15.47)
  even slightly below PRE-STEP-1 (15.77), proving the per-step defensive
  sweep at line 3322 also reclaims any residue.

- **Per-step wall clock at the production envelope.** v10f averaged
  ~200s/step across 3 steps. Breakdown (averaged, G=4 NSTEPS=3 FBS=1):

  | Phase | Time | Share |
  |------|------|------|
  | gen (vLLM HTTP)        | ~5.6 s   | 3% |
  | grpo (fwd + bwd + helper) | ~108.9 s | 54% |
  | &nbsp;&nbsp;– fwd      | ~3.8 s | |
  | &nbsp;&nbsp;– bwd      | ~5.6 s | |
  | &nbsp;&nbsp;– **v9 release helper** | **~99 s** | |
  | clip                   | ~0.1 s | <1% |
  | optimizer (AdamWBf16)  | ~85.8 s | 43% |

  ~93% of step time is CPU-side workarounds (gloo bounce + CPU
  optimizer). Generation is already negligible.

## Production envelope

Single-node EP=8 dp_replicate=1, current code:

- Optimizer **must be** `torchtune.dev.bioreason.optim.AdamWBf16`
  (yaml comment line 105 already lists it as the contingency).
- `forward_batch_size=1` is required to exercise the chunked-accumulate
  path under `G≥2`. `fbs=2` collapses `G=2` into a single chunk and only
  exercises activation OOM, not the chunked path the helper unblocks.
- Validated up to `G=4 NSTEPS=3` with finite losses and PRE-STEP flat.
  Per-step is ~3:20.

## What v10 does **not** prove

- `dp_replicate ≥ 2` end-to-end with the helper — the existing v75 XCCL
  cross-replica sync at line 2945 has not been retested with real grads
  in EP mode. (Single-replica was the only way to fit on the available
  capacity hold today.)
- Step-time stability at full production `G≥8`. The two big costs (helper
  ~99s and AdamWBf16 ~86s) are essentially constant per step regardless
  of G, so wall time should remain ~200s — but per-step PRE-STEP rise
  with larger activation pressure is unmeasured.
- Loss quality: G=4 with `max_gen=32` is still in the regime where many
  responses get truncated before any reward signal. v10b2 / v10e / v10f
  all show finite-but-small losses (0.006–0.009). Reward curve has not
  been run.

## Comparison with EP=1 (replicated experts) production

| Mode | Per-step | Status |
|------|----------|--------|
| EP=1 single-node G=8 (validated 2026-04-27) | 54.8 s | production |
| EP=8 dp_replicate=1 G=4 (v10f) | ~200 s | proof-of-concept |

EP=8 is currently **3.6× slower per step** than the validated EP=1 path,
even at half the rollout count. 93% of EP=8 wall time is the v9 helper
gloo bounce and the CPU AdamWBf16 walk — both are workarounds for the
v59 ordering race and the L0 headroom ceiling, not fundamental to EP. A
GPU-resident reduce-scatter (XCCL on-device) and a return to GPU
optimizer state are both feasible if the v59 race can be avoided some
other way.

## Next directions (not yet committed)

A. **3-node EP=8 dp_replicate=3 / dp_shard=8.** Tests whether v75 XCCL
   cross-replica sync works with the real grads the helper now produces,
   and whether spreading optimizer state across 3 replicas restores
   enough headroom to use plain AdamW back on device. Matching capacity
   was not held during the v10 series.

B. **Rewrite `_ep_release_fsdp_unsharded_grads` to use XCCL reduce-scatter
   on-device.** Currently the helper does CPU gloo bounce because the v59
   race was originally diagnosed in XCCL. Re-test whether the global
   barrier alone is sufficient ordering with on-device XCCL.

C. **Activation profile of larger G.** Sweep G=8/16 at NSTEPS=1 with the
   helper + AdamWBf16. PRE-STEP-1 already had ~7 GiB headroom in v10f at
   G=4; G=8 may still fit before any further intervention.
