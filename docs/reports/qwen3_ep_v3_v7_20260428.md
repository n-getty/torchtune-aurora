# Qwen3-30B-A3B EP=4/DP=3 — v3-v7 Investigation (2026-04-28)

PBS job 8453205 (debug-scaling, 2 nodes, 1h walltime). Continuation of v1/v2
(job 8453156) — see `docs/features/moe_integration.md` "Qwen3-30B-A3B Expert

## Goal

Produce the first `loss=` line for Qwen3-30B-A3B under EP=4/DP=3.

## Topology

- Node 0: vLLM 3×TP=4 server (`run_qwen3_30b_vllm_server.sh`)
- Node 1: torchtune training, 12 tiles EP=4/DP=3 (`run_qwen3_30b_ep4_vllm_2node.sh`)
- Wrapper: `experiments/ep_parallelism/hold_qwen3_ep_v{3-7}.sh`

## Per-run timeline

### v3 — Tier 1 environment fix
**Change**: `CCL_ZE_CACHE_OPEN_IPC_HANDLES_THRESHOLD=65536`, `unset XPU_USM_ALLOC_SO`,
`PYTORCH_ALLOC_CONF=garbage_collection_threshold:0.99`.

**Result**: Crashed at op #425 (train fwd, layer ~13), `banned:1` PDE write fault.
Gloo coll= flat at ~10s/layer (was monotonically growing 19s→134s in v2).
**Conclusion**: Tier 1 env vars fix gloo CPU pressure but do NOT fix the GPU PDE.
The v2 escalating coll= was caused by `XPU_USM_ALLOC_SO` leaking from the held
shell — the launcher must explicitly `unset` it.

### v4 — XPU sync after H2D in EP collectives
**Change**: Added `torch.xpu.synchronize()` after the gloo→XPU H2D copy in both
`_ep_reduce_scatter` and `_ep_all_gather` in `torchtune/modules/moe/_parallelism.py`.

**Result**: Same crash (op #427, `banned:1` PDE). coll= escalated back to ~40s
(syncs added gloo pressure).
**Conclusion**: Async H2D use-after-free disproven. Reverted in v5.

### v5 — disable forward chunking
**Change**: `EXTRA_OVERRIDES='forward_batch_size=4'` (config has
`forward_batch_size=2 grpo_samples=4`, so fbs=4 ⇒ single fwd, no chunking).

**Result**: Clean Python OOM in pass 2 at 60.83 GiB. Ranks 3/7/11 (last in each
EP group) hit OOM first, confirming expert-load imbalance.
**Conclusion**: The `with torch.no_grad():` exit transition is NOT the trigger
(crash mode changed entirely under no chunking). Memory is the proximate problem
in this config but doesn't explain v3/v4's banned:1.

### v6 — halve max_generated_tokens
**Change**: `EXTRA_OVERRIDES='max_generated_tokens=64'` (was 128 in config).
Restored `forward_batch_size=2` (chunked).

**Result**: First time past v3's op #425. Reached op #439 (train fwd, layer ~27/48),
`banned:0` PDE.
**Conclusion**: Crash-position correlated with activation size. Looked like a
clean memory-pressure story.

### v7 — quarter activations
**Change**: `EXTRA_OVERRIDES='max_generated_tokens=32 grpo_samples=2'`. Combined
2× shorter responses with 2× fewer rollout samples → 4× total activation
reduction vs v3.

**Result**: Crashed at op #253-equivalent (train fwd, layer ~30 in equivalent
terms — grpo_samples=2 changes the op-numbering because there are 4 fwd passes
instead of 5). `banned:1` PDE at GPU address `0xff0100059cf2d000`.
**Conclusion**: 4× memory reduction yielded only ~3 more layers vs v6's 2×
reduction (layer 27 → 30). With proportional scaling the expectation was layer
~41. **Pure activation-memory pressure is disproven.**

## Crash anatomy

All v3/v4/v6/v7 crashes share:
- Location: train fwd (the only fwd pass with autograd save)
- Address range: `0xff01...` — L0 IPC handle range, not the standard USM
  device range (`0xff00...`)
- Signature: `Segmentation fault from GPU at 0xff... type: 0 (NotPresent),
  level: 1 (PDE), access: 1 (Write)`
- The `banned:1` vs `banned:0` differ only in whether the kernel marks the
  context for migration; both are the same fault class

ref/policy fwds (no_grad, same EP code path, same gloo bounce) succeed every
time. The difference is purely **autograd graph construction over the EP
collectives during train fwd**.

## What we ruled out

| Hypothesis | Disproved by |
|------------|--------------|
| Async H2D use-after-free in EP collectives | v4: `torch.xpu.synchronize()` after H2D didn't help |
| `with torch.no_grad():` → grad transition | v5: chunking off changed crash mode to clean Python OOM |
| Pure activation memory pressure | v7: 4× reduction = +3 layers (vs proportional ~14) |
| Saved-tensor pinning of gloo bounce-CPU buffers | Code review: `_AllGatherRS`/`_ReduceScatterAG` save only `ctx.group`, no tensors |
| Gloo CPU-bounce TCP backpressure | v3: Tier 1 env collapsed coll= to flat 10s; PDE persisted |

## Working hypothesis

L0 IPC handle pressure during autograd graph construction over EP collectives.
- 48 MoE layers × 2 EP collectives × 5 fwd passes per step = ~480 IPC
  handle registrations per step
- `CCL_ZE_CACHE_OPEN_IPC_HANDLES_THRESHOLD=65536` helps but is apparently
  insufficient when autograd is live (each fwd's IPC handles must remain
  registered until the corresponding bwd consumes them)
- ref/policy fwds release their IPC handles immediately (no_grad → no graph
  → no held references), so they never accumulate

## v8 plan (for next allocation)

**Force the EP gloo PG to TCP-only transport** to eliminate SHM IPC handles
entirely. Suspect knobs:
- `GLOO_DEVICE_TRANSPORT=tcp`
- `GLOO_USE_LIBUV=1`
- Or explicit: `dist.new_group(..., backend="gloo",
  pg_options=ProcessGroupGloo.Options(devices=[create_tcp_device]))` for the
  EP PG only

If TCP-only also crashes:
- The IPC pressure is upstream (FSDP2 + CCL share the L0 driver IPC pool, EP
  collectives are just the load that exceeds the threshold)
- Next: reduce EP collective count by selectively disabling AC on a subset of
  MoE layers (each fewer-AC layer = 1 less fwd-pass-worth of IPC handles)
